# 캐스케이드 바디-헤드 탐정단
## 중학생도 이해하는 `cascaded_body_head_model.py` 완전 해설

- 대상 독자: affine + MNIST, 기본 convolution, 간단한 torch 구성 경험이 있는 학습자
- 구성: 20장 + 부록(실험/디버깅/미니 프로젝트)

---

## 용어집(Glossary)
- `slot/query`: 고정 길이 예측 단위. 본문에서는 `slot/query`로 통일한다.
- `objectness/logit/probability`: head 출력 원값은 `logit`, `sigmoid(logit)` 값은 `probability`로 표기한다.
- `normalized/global/feature coordinates`: `normalized/global`은 이미지 기준 `0~1` 좌표, `feature coordinates`는 feature map 스케일 좌표를 뜻한다.
- `ROI/ROI Align`: `ROI`는 관심 영역 박스, `ROI Align`은 ROI에서 정렬 샘플링으로 특징을 추출하는 연산을 뜻한다.
- `matched/negative`: `matched`는 GT와 매칭된 양성 `slot/query`, `negative`는 매칭되지 않은 음성 `slot/query`를 뜻한다.

# 19장. 디버깅 체크리스트

## 학습 목표
- 이 장의 핵심 개념을 1문장으로 설명할 수 있다.
- 입력/출력 텐서와 핵심 연산 흐름을 따라갈 수 있다.

## 이야기로 시작
앞 장의 개념을 다음 단계 코드 동작과 연결해 이해하는 장이다.

## 코드 산책
핵심 함수의 입력 -> 중간 텐서 -> 출력 순서로 읽으며 shape와 좌표 단위를 확인한다.

## 레이어 왜 쓰나
이 구성은 계산량, 학습 안정성, 디버깅 용이성을 함께 맞추기 위한 선택이다.

## 작은 실험
- 기본 shape 출력 1회
- 핵심 파라미터 1개 변경 후 로그 비교
- 실패 시 입력 범위/매칭 상태 점검

## 한 줄 수학
핵심식: 입력 벡터(또는 박스) -> 변환 함수 -> 목적 좌표/점수로 매핑한다.

## 실전 디버깅 순서(10분 루틴)
1. `한 배치 고정`부터 시작한다. 매 스텝 입력이 바뀌면 원인 추적이 어려워진다.
2. `stage별 최소 로그`를 찍는다. backbone feature 통계, body/head 박스 범위, matching 수, 손실 항 기여도를 같은 스텝에서 같이 본다.
3. `숫자 + 시각화`를 같이 본다. loss 숫자만 보면 정상처럼 보이는 실패 케이스가 많다.

```python
# 1 step 디버그 로그 예시(필요 키만 골라 출력)
def scalar(v):
    return float(v.detach().cpu()) if hasattr(v, "detach") else float(v)

with torch.no_grad():
    outputs = model(images)
    pred_body = outputs["pred_body_boxes"]      # [B, Q, 4], normalized cxcywh
    pred_head = outputs["pred_head_boxes"]      # [B, Q, 4], normalized cxcywh
    obj_logit = outputs["pred_objectness_logits"]  # [B, Q]
    obj_prob = obj_logit.sigmoid()

print("body[min,max]=", scalar(pred_body.min()), scalar(pred_body.max()))
print("head[min,max]=", scalar(pred_head.min()), scalar(pred_head.max()))
print("obj_prob[mean]=", scalar(obj_prob.mean()))
```

## Stage별 진단 체크리스트 + 실패 예시

### 1) Backbone
- 체크: 입력 정규화 범위(`0~1` 또는 mean/std)가 학습 코드와 추론 코드에서 같은지 확인한다.
- 체크: backbone 출력 feature의 `mean/std`가 극단적(`std`가 거의 0, 또는 매우 큼)인지 본다.
- 체크: `model.train()`/`model.eval()` 상태가 의도대로인지 확인한다(BN/Dropout 영향).
- 실패 예시: `objectness`가 처음부터 끝까지 `0.5` 근처에 붙어 있으면 backbone 출력이 거의 상수일 가능성이 크다.

### 2) Body(Stage1 box)
- 체크: `pred_body_boxes`가 정규화 범위(대략 `0~1`)를 벗어나는지 본다.
- 체크: `cxcywh -> xyxy` 변환 후 `x1 <= x2`, `y1 <= y2`가 깨지는 샘플 비율을 기록한다.
- 체크: 폭/높이(`w/h`) 평균이 너무 작아(`0` 근처) ROI가 사실상 점처럼 되는지 확인한다.
- 실패 예시: body 박스가 화면 한쪽 구석에만 몰리면 sigmoid/클램프 위치 또는 좌표 해석 순서를 먼저 의심한다.

### 3) ROI/ROI Align
- 체크: ROI 좌표 단위가 feature map 스케일인지(정규화 전역 좌표를 그대로 넣지 않았는지) 확인한다.
- 체크: `rois[:, 0]`(batch index)이 현재 batch 구성과 맞는지 확인한다.
- 체크: ROI Align 출력이 전부 0이거나 모든 slot에서 거의 동일하면 좌표 스케일/순서 오류일 가능성이 높다.
- 실패 예시: head loss만 계속 정체되면 ROI crop이 잘못돼 같은 feature만 반복 추출되는 경우가 많다.

### 4) Head(Stage2 box)
- 체크: head 상대좌표 디코딩이 body 기준으로 올바르게 적용되는지(원점/스케일) 확인한다.
- 체크: flatten 전후 shape이 기대와 같은지 확인한다(샘플 축/slot 축 섞임 방지).
- 체크: `pred_head_boxes`가 body 영역과 완전히 무관한 위치로 튀는 샘플 비율을 본다.
- 실패 예시: head 박스가 항상 body 중심으로만 나오면 디코딩 수식에서 상대폭/높이 반영이 누락됐을 수 있다.

### 5) Matching
- 체크: `matched_count`, `negative_count`, `matched_ratio`를 매 스텝 로그로 본다.
- 체크: IoU 임계값이 과도하게 높아 양성 매칭이 거의 0이 되지 않는지 확인한다.
- 체크: GT 좌표 형식(`cxcywh` vs `xyxy`)이 예측과 맞는지 재검증한다.
- 실패 예시: `kept_slots > 0`인데 `matched_count = 0`이 반복되면 좌표 변환 불일치나 임계값 설정이 우선 의심 지점이다.

### 6) Loss
- 체크: 총 loss만 보지 말고 `objectness/body/head/iou` 항을 분리해 본다.
- 체크: `가중치 x 평균손실` 기여도로 어떤 항이 학습을 지배하는지 계산한다.
- 체크: 특정 항이 `NaN/Inf`로 튀는 스텝의 입력/예측 범위를 함께 저장한다.
- 실패 예시: 총 loss는 내려가는데 품질이 나쁘면 objectness 항만 과도하게 최적화된 불균형 상황일 수 있다.

```python
# 매칭/손실 빠른 점검 예시
loss_dict, aux = criterion(outputs, targets, return_aux=True)
matched = int(aux["matched_count"])
negative = int(aux["negative_count"])
ratio = matched / max(matched + negative, 1)

print("matched=", matched, "negative=", negative, "ratio=", round(ratio, 4))
for k, v in loss_dict.items():
    print(k, float(v.detach().cpu()))
```

## 빠른 진단표: if X happens, check Y
| if X(증상) | check Y(즉시 확인) |
|---|---|
| objectness 확률이 계속 0.5 근처 | backbone feature `std`, 입력 정규화, BN/Dropout 모드 |
| body 박스가 이미지 밖으로 자주 벗어남 | sigmoid/클램프 위치, 좌표계(`cxcywh/xyxy`) 변환 순서 |
| ROI feature가 거의 전부 0 | ROI 좌표 스케일(전역 vs feature), `rois[:,0]` batch index |
| head 박스가 body와 무관하게 튐 | head 디코딩 식의 body 기준 원점/스케일 적용 여부 |
| `matched_ratio`가 계속 0 | IoU 임계값, GT/예측 좌표 형식 일치 여부 |
| 총 loss만 감소하고 품질은 정체 | 손실 항별 기여도(`가중치 x 평균손실`) 재점검 |

## 오해/바로잡기
- 오해: 총 loss만 내려가면 학습은 정상이다. 바로잡기: matched_ratio, kept_slots, 시각화 결과를 같이 봐야 실패 모드를 조기에 찾을 수 있다.
